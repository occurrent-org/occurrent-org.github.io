Occurrent 0.31.0 is released. Where 0.30.0 added Dynamic Consistency Boundary support, this release is about higher-level DSLs and declarative annotations across the event-sourcing lifecycle. It adds a Saga DSL for event-driven process managers, a Projection DSL with a `@Projection` annotation for read models, first-class snapshots, synchronous subscriptions that update a read model on the write path, and pluggable projection event sources so a projection can be fed from a broker instead of a MongoDB change stream. It builds on 0.30.0 and requires Java 21.

Most of the release is new modules, so there is nothing to migrate for those. Three existing APIs moved, all of them 0.30.0 coordinates or types, and an OpenRewrite recipe rewrites every reference. Read the [backward-incompatible changes](#backward-incompatible-changes) below before upgrading.

## Highlights

### A Saga DSL for event-driven process managers

A saga reacts to events, and to their absence (timeouts), by issuing commands while holding per-instance state. It is the command-issuing dual of a `Decider`, pure data and pure functions with no I/O, so it unit-tests the same way a decider or a view does. A reaction decides its effects as data (issue a command, start or cancel a timer) against the folded state, which keeps replay effect-free. Timers carry a `Duration` or an `Instant` rather than a clock reading, so a reaction stays deterministic in tests, and they live in the saga's own state envelope, which makes timer bookkeeping exactly-once because it is saved atomically with the state.

There are two authoring surfaces that compile to the same descriptor. The machine-core builder (`Saga.builder(...)` in Java, `saga(initialState) { }` in Kotlin) is the low-level fold-and-react form. The linear flow layer (`FlowSaga.builder()` in Java, `saga { }` in Kotlin) has steps, branches, joins, and per-step timeouts that read like the process itself. Correlation is per-event-type with a `correlateAll` fallback, validated at build time so a handled event can never lack a correlation at runtime. `@Saga` declares one on the blocking Spring stack and registers it through the same catch-up and durable infrastructure `@Projection` uses, with a lease-gated timer poller so only one instance in a cluster fires a given timeout.

A saga is not a substitute for a Dynamic Consistency Boundary. Where two rules must hold in one append, DCB is the right tool. A saga is for genuinely cross-boundary, time-involving, eventually-consistent processes. A new `order-fulfillment` example module demonstrates both surfaces, in-memory and Docker-free. See the [saga documentation](/documentation#sagas) for the walkthrough. Resolves [#124](https://github.com/johanhaleby/occurrent/issues/124).

### A Projection DSL and `@Projection` read models

A `Projection` couples a `View` (the pure fold) with the function deriving which view instance an event updates and the event types it handles, and a `DcbProjection` adds a DCB read boundary. You describe the fold with a type-safe, per-event-type handler builder, and the registered handlers also determine the subscription filter, so the event types are declared once. A handler and the id function can read the event's metadata (stream id and version, global position, CloudEvent extensions) through optional metadata-carrying overloads, so a projection can key a view instance by stream id or fold on the global position.

`@Projection` on a factory method returning a `Projection` or `DcbProjection` registers it as a managed subscription on both the blocking and reactor stacks, through the same catch-up and durable-checkpoint infrastructure `@DcbSubscription` and `@StreamSubscription` already use. Materialization is store-agnostic through `ViewStateRepository`, `MaterializedView`, or a Spring Data `CrudRepository`. `mode` selects `ASYNC` (catch-up then live) or `SYNCHRONOUS` (read-your-writes). A new `projection-dsl` example module demonstrates it in Java and Kotlin for both stream and DCB. See the [projection documentation](/documentation#projections). Resolves [#194](https://github.com/johanhaleby/occurrent/issues/194).

### First-class snapshots

A snapshot is an opt-in optimization that lets a command replay fold only the events written after a saved state instead of the whole history. It is a discardable, schema-versioned cache of a fold result at a known version, so a changed state shape or a missing snapshot falls back safely to a full replay, and a snapshot is never a source of truth. It works from a `Decider` and from a plain `View`, across stream and DCB and across the blocking and reactor stacks. The `@Snapshot` annotation declares and maintains one through the same infrastructure `@Projection` uses, and a `SnapshotPolicy` unifies technical and domain triggers (`everyNEvents`, `whenState`, `whenTerminal` for closing the books, and more). Persistence through the DSL executors is best-effort by default, saved after the write commits, so a lost snapshot only costs a fuller replay and never fails an already-committed command. A new `closing-the-books` example demonstrates both a technical every-N snapshot and a domain period close. See the [snapshot documentation](/documentation#snapshots).

### Synchronous subscriptions

A subscription can now be declared synchronous with the new `@SynchronousSubscription` annotation, or by registering a handler on a `SynchronousSubscriptionModel` through the `Subscriptions` DSL with no Spring required. The application service invokes it on the writer thread, before `execute` returns, so a projection can be updated in the write path for read-your-writes. This is a distinct mechanism from the existing asynchronous subscriptions, which are unchanged. Transactions are opt-in: configure a `TransactionExecutor` on the application service to make the write and the handlers commit atomically, with a Spring-backed or a native MongoDB `ClientSession`-backed executor. Available on the blocking and reactive, stream and DCB application services.

### Pluggable projection event sources

The Projection DSL can now be fed from an external source instead of a MongoDB change stream. `PushSubscriptionModel` is a register-only `Subscribable` driven by an `accept(CloudEvent)` call, so a RabbitMQ, Kafka, Spring-event, or HTTP listener can drive projections in production. Occurrent stays transport-neutral, adding no broker dependency. A `CatchupThenPushSubscriptionModel` adds a one-time catch-up in front of the live feed, replaying a projection's history from the event store on first subscribe and then handing over to the broker. Declaratively, `@Projection` gains a `source` attribute, `source = Source.PUSH`, that binds a projection to a push model and wraps it in the catch-up automatically. For a source that already hands you domain events, `Projections.domainEventFeed(...)` folds a domain event straight into the read model with no CloudEvent conversion on the live path. See [ADR 62](https://github.com/johanhaleby/occurrent/blob/main/doc/architecture/decisions/0062-pluggable-projection-event-source.md) for the design and the "broker owns live-resume" contract.

### A reusable command-dispatch capability

The command-dispatch machinery, previously internal to the Saga DSL, is now a standalone `occurrent-command-dispatch` module so any command producer can use it. `CommandDispatcher<C>` is the producer-facing port that issues a command, and `StreamIdResolver<C>` is a named command-to-stream-id function, the stream-side counterpart of the DCB `TagGenerator`. A new `@TargetStreamId` annotation marks the command member holding the target stream id, and `occurrent-command-dispatch-annotation` reads it reflectively, mirroring the DCB `@DcbTag` and `AnnotationTagGenerator`. The blocking Spring Boot starter contributes a default resolver bean, overridable with your own.

### Smaller additions

- The view, projection, and saga DSLs can now fold with event metadata (stream id and version, global position, CloudEvent extensions), reusing the same `EventMetadata` a subscriber already sees.
- Java-friendly facades for running a `Decider` or `DcbDecider` through an application service, giving Java callers the one-call `execute(command, decider)` ergonomics the Kotlin extensions already had.
- Nicer Kotlin DCB query criteria from event classes, `type<OrderPlaced>()` and `types<OrderPlaced, OrderCancelled>()` with the base type inferred, in place of `::class.java`.
- DCB reads can be capped and directed with `DcbReadOptions`. `backwardsLimited(1)` reads the single highest-position event matching a criteria in one round trip, so a gapless business sequence can look up its last entry without folding the whole history.
- A subscription handler can receive the stream id or version directly through `@StreamId` and `@StreamVersion` parameter annotations, without declaring an `EventMetadata` parameter.
- The MongoDB change stream backing a subscription can be tuned for throughput and latency with `batchSize(int)` and `maxAwaitTime(Duration)`. Both are opt-in and default to unset. Resolves [#173](https://github.com/johanhaleby/occurrent/issues/173).

## Backward-incompatible changes

These three changes affect a 0.30.0 caller. Each is either a coordinate rename or a package move, none changes stored data, and the `org.occurrent.UpgradeToOccurrent_0_31` OpenRewrite recipe rewrites every affected reference for you. Start with the [upgrade guide](https://github.com/johanhaleby/occurrent/blob/main/doc/migration/upgrading-to-0.31.0.md) for the recipe setup.

| Change | What changed | How to upgrade |
|--------|--------------|----------------|
| Annotation enums hoisted | `ResumeBehavior`, `StartupMode`, `StartPosition`, `Capability`, and `Mode` move out of the individual annotations, where each declared its own copy, and become shared top-level types in `org.occurrent.annotation`. `@DcbSubscription`'s `DcbStartPosition` becomes the shared `StartPosition`. `@StreamSubscription` keeps its own `StartPosition` because its `BEGINNING_OF_TIME` constant genuinely differs. | Run the recipe. It rewrites a nested reference such as `Subscription.ResumeBehavior.DEFAULT` to `ResumeBehavior.DEFAULT`. |
| Checkpoint-storage modules renamed | The four checkpoint-storage modules are renamed from `-position-storage` to `-checkpoint-storage`, so the coordinate matches the `CheckpointStorage` type it ships (finishing the `SubscriptionPosition` to `Checkpoint` rename 0.30.0 applied to the types). The groupId, packages, and classes are unchanged, so it is a coordinate-only change. | Run the recipe. It rewrites the coordinates for Maven and Gradle, or remap them by hand from the table in the upgrade guide. |
| `EventMetadata` relocated | `EventMetadata` moves from `org.occurrent.dsl.subscription` to `org.occurrent.cloudevents` (module `cloudevents-extension`), now that it is the shared fold currency for the saga, projection, and view DSLs. It is rewritten from a Kotlin `data class` to a plain Java class, dropping its Kotlin-only surface (reified `get`, operator `get`, `copy`), which was essentially unused. The typed accessors are unchanged in name and behavior. | Run the recipe. It updates the import and every reference. |

Full details are in the [changelog](https://github.com/johanhaleby/occurrent/blob/main/changelog.md), and the [documentation](/documentation) has been updated with the new features and examples.
